# JOIN 为什么慢？

## 问题

`JOIN` 为什么会慢？为什么有时即使有索引，连接查询仍然不快？

## 我的理解

`JOIN` 本质上可以拆成三步：

1. 过滤：先根据 `WHERE`、子查询或连接前置条件减少数据量。
2. 匹配：按照 `JOIN` 条件在两张表或多张表之间寻找能匹配的行。
3. 拼接：把匹配到的行组合成最终结果集。

慢通常卡在“匹配”这一步。匹配需要在表之间反复查找、比较和组合数据；如果参与匹配的数据量很大，或者匹配条件不能高效利用索引，成本就会快速上升。

## 常见 JOIN 算法

不同数据库实现细节不同，但常见 `JOIN` 算法大致可以理解为下面几类。

### Nested Loop Join

嵌套循环连接可以理解为：先从外层表取一行，再去内层表找匹配行；外层表有多少行，就重复多少次内层匹配。

简化理解：

```text
for outer_row in outer_table:
    for inner_row in inner_table:
        if outer_row.key = inner_row.key:
            output joined row
```

如果外层结果很小，这种方式可以很快；如果外层结果很大，内层又不能快速定位，就会变成大量重复扫描。

适合场景：

- 外层表过滤后结果很小。
- 内层表很小。
- 查询只需要少量匹配结果。

容易变慢的场景：

- 外层行数很多。
- 内层没有合适索引。
- 每次内层查找都要扫描大量数据。

### Index Nested Loop Join

索引嵌套循环连接是在嵌套循环的基础上，内层表通过索引查找匹配行。

简化理解：

```text
for outer_row in outer_table:
    use index on inner_table.join_key to find matched rows
```

这也是索引对 `JOIN` 很重要的原因：它把“每次扫内层表”变成“每次通过索引定位内层数据”。

适合场景：

- 外层结果不大。
- 内层连接字段有高效索引。
- 连接条件是等值条件。

容易变慢的场景：

- 外层结果太大，导致索引查找次数过多。
- 内层索引选择性差，一次查找命中很多行。
- 命中后需要大量回表。

### Hash Join

哈希连接通常会先选择一侧表构建哈希表，再扫描另一侧表，用连接键去哈希表里找匹配项。

简化理解：

```text
build hash table from smaller input by join_key
for row in larger input:
    probe hash table by row.join_key
```

哈希连接通常适合等值连接，尤其适合两边数据量都不小、单纯靠索引反复查找成本较高的场景。

适合场景：

- 等值连接。
- 两侧输入都比较大。
- 可以先把较小一侧放进内存构建哈希表。

容易变慢的场景：

- 构建哈希表的一侧太大，内存放不下。
- 哈希表溢出到磁盘。
- 数据分布倾斜，某些 key 匹配行特别多。
- 非等值连接通常不适合普通哈希连接。

### Sort Merge Join

排序合并连接会先把两侧数据按连接键排序，然后像归并两个有序数组一样向前匹配。

简化理解：

```text
sort left input by join_key
sort right input by join_key
merge two sorted inputs and output matched rows
```

如果两边本来就能按索引顺序读出，排序成本可能较低；如果需要额外大排序，成本就会上升。

适合场景：

- 两侧数据已经按连接键有序。
- 连接结果较大。
- 范围条件或有序输出能复用排序结果。

容易变慢的场景：

- 两边都需要大规模排序。
- 排序内存不足，溢出到磁盘。
- 连接前没有先过滤，导致排序数据量过大。

## 算法与优化思路的关系

理解这些算法后，`JOIN` 优化方向会更清楚：

- 减少外层输入行数，可以降低 Nested Loop 的循环次数。
- 给内层连接字段建索引，可以把普通 Nested Loop 变成更高效的 Index Nested Loop。
- 先过滤小结果，可以让 Hash Join 的哈希表更小，也能减少 Sort Merge Join 的排序量。
- 避免函数和类型转换，是为了让连接条件保留索引查找、哈希匹配或排序匹配能力。
- 避免一对多、多对多结果膨胀，是因为算法再高效，也挡不住最终结果本身变得巨大。

## 常见变慢原因

### 数据量太大

即使有索引，参与 `JOIN` 的数据量很大时也不一定快。

原因是索引只能帮助定位数据，但定位之后仍然可能产生大量读取、回表、比较和结果拼接。如果命中行数过多，优化器也可能认为走索引不如其他访问方式便宜。

### JOIN 字段没有索引

如果连接字段没有索引，数据库在匹配时可能需要扫描大量数据。

例如：

```sql
SELECT *
FROM orders o
JOIN users u ON o.user_id = u.id;
```

这类查询通常需要关注：

- `orders.user_id` 是否有索引。
- `users.id` 是否是主键或唯一索引。
- 哪张表先被过滤，哪张表被反复查找。

### JOIN 条件破坏索引或匹配能力

如果在连接字段上使用函数、表达式或发生隐式类型转换，可能会破坏索引使用。

例如：

```sql
JOIN users u ON CAST(o.user_id AS CHAR) = u.id
JOIN users u ON LOWER(o.email) = u.email
```

这类写法会让数据库难以直接使用索引中的原始有序值完成匹配。

### 非等值连接

等值连接通常更容易优化。非等值连接需要结合数据分布判断成本。

例如：

```sql
JOIN price_rule r
  ON p.price >= r.min_price
 AND p.price < r.max_price
```

这种连接可能涉及范围匹配。如果范围重叠多、命中比例高，匹配成本会明显上升。

### 多表 JOIN 顺序不当

多表 `JOIN` 时，中间结果大小非常关键。理想情况是先过滤出小结果，再参与后续连接，让中间结果尽可能小。

如果先连接大表，或者过滤条件太晚生效，数据库可能需要处理大量中间结果。

### 一对多或多对多导致结果集膨胀

`JOIN` 不只是查找，还会生成组合结果。

如果一条主表记录能匹配多条明细记录，结果集会变大；如果是多对多关系，结果集可能膨胀得更明显。

这会增加：

- 匹配成本。
- 排序、分组、去重成本。
- 网络传输成本。
- 应用层处理成本。

### SELECT * 增加读取和传输成本

`SELECT *` 会把不需要的字段也查出来。

在 `JOIN` 场景下，它可能带来：

- 更多回表。
- 更宽的中间结果。
- 更多内存占用。
- 更多网络传输。

## 优化核心

`JOIN` 优化的核心是减少参与 `JOIN` 的数据，并让匹配过程更容易利用索引。

## 参考方向

### 先过滤，再 JOIN

优先让过滤条件尽早生效，减少进入连接的数据量。

```sql
SELECT o.id, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.created_at >= '2026-06-01'
  AND o.status = 'PAID';
```

如果过滤条件复杂，也可以考虑先用子查询或临时结果缩小范围，再连接。

### 在连接方向上建立合适索引

索引要服务实际连接方向和过滤条件。

常见关注点：

- 被连接查找的一侧是否有索引。
- 外键字段是否有索引。
- 过滤字段和连接字段是否需要组合索引。
- 联合索引顺序是否匹配过滤和连接条件。

### 只查需要的字段

避免 `SELECT *`，只返回当前业务真正需要的字段。

```sql
SELECT o.id, o.created_at, u.name
FROM orders o
JOIN users u ON o.user_id = u.id;
```

### 避免在 JOIN 条件上做函数和类型转换

尽量让连接字段类型一致，直接用原始列值匹配。

不推荐：

```sql
JOIN users u ON CAST(o.user_id AS CHAR) = u.id
```

更推荐：

```sql
JOIN users u ON o.user_id = u.id
```

前提是两边字段类型本来就设计一致。

## 当前结论

`JOIN` 慢不是因为 `JOIN` 本身一定慢，而是因为参与匹配的数据太多、匹配条件不适合索引、连接顺序导致中间结果膨胀，或者最终拼接出的结果集太宽太大。

优化时优先问：

- 能不能先过滤掉更多数据？
- 连接字段有没有合适索引？
- `JOIN` 条件有没有函数、表达式或隐式转换？
- 多表连接时中间结果是否最小？
- 是否发生一对多、多对多结果集膨胀？
- 是否查了太多不需要的字段？
