# WHERE 条件怎么影响索引使用？

## 问题

`WHERE` 条件为什么有时能很好地利用索引，有时却会导致索引失效，或者让优化器放弃索引？

## 我的理解

索引本质上是数据库额外维护的一份有效数据结构。以 B+ Tree 为例，索引里大致保存的是“索引列的值 + 指向原表数据行的位置”。要让索引真正加速查询，查询条件必须能利用这份有序结构快速缩小扫描范围。

但即使条件看起来能用索引，最终是否使用索引也不是绝对规则，而是由数据库优化器按成本决定。优化器会判断：这里走索引是否比不走索引更便宜？

这个判断依赖数据库维护的统计信息，例如：

- 表大概有多少行。
- 某列有多少不同值。
- `NULL` 比例是多少。
- 哪些值最常见。
- 常见值各占多少比例。
- 范围分布大概是什么样。
- 数据物理顺序和索引顺序是否相关。

基于这些统计信息，优化器才能估算某个查询条件大概会命中多少行、会覆盖多大范围、走索引是否值得。

## 单列索引与单条件

如果是单列索引，并且 `WHERE` 中是简单的等值或范围条件，索引通常比较容易被利用。

例如：

```sql
WHERE user_id = 1001
WHERE created_at >= '2026-06-01'
```

这类条件可以直接在索引结构上定位或扫描一段范围。

## 函数和表达式

如果对索引列使用函数或表达式，通常会破坏索引的直接使用。原因是索引里存的是原始列值，而查询条件比较的是计算后的值，数据库可能无法直接用索引中的有序值定位。

例如：

```sql
WHERE DATE(created_at) = '2026-06-02'
```

更适合索引的写法通常是：

```sql
WHERE created_at >= '2026-06-02 00:00:00'
  AND created_at <  '2026-06-03 00:00:00'
```

隐式类型转换也有类似影响。如果列类型和条件值类型不一致，数据库可能需要先转换再比较，从而影响索引使用。

## LIKE 条件

`LIKE` 是否能利用索引，通常取决于通配符位置。

可以较好使用索引的形式：

```sql
WHERE name LIKE '张%'
```

因为条件仍然能从固定前缀开始定位索引范围。

较难使用普通 B+ Tree 索引的形式：

```sql
WHERE name LIKE '%张'
WHERE name LIKE '%张%'
```

因为没有固定开头，数据库无法从索引有序结构中直接定位起点。

## 联合索引

联合索引通常是按字段顺序组织的辅助结构。对于符合索引字段顺序的查询条件，可以较好利用索引。

例如联合索引：

```text
(tenant_id, status, created_at)
```

较容易利用：

```sql
WHERE tenant_id = ?
  AND status = ?
  AND created_at >= ?
```

利用率有限：

```sql
WHERE status = ?
WHERE created_at >= ?
```

因为没有使用前面的字段，数据库难以直接按照联合索引的整体顺序定位。

## OR 条件

`OR` 条件可能让优化器认为走索引不划算，进而改成全表扫描。

原因包括：

- 多个条件可能涉及不同索引，需要合并结果。
- 某些分支选择性很低，会命中大量数据。
- 合并多个索引结果的成本可能高于直接扫描。

所以 `OR` 不是一定不能用索引，而是更容易受到成本判断影响。

## 不等条件与 NULL

`<>`、`!=`、`IS NULL`、`IS NOT NULL` 这类条件是否使用索引，取决于数据分布和命中比例。

例如：

- 如果 `NULL` 很少，`IS NULL` 可能适合走索引。
- 如果 `NULL` 很多，走索引可能不如全表扫描。
- 如果 `<>` 会命中绝大多数行，优化器也可能放弃索引。

## 当前结论

`WHERE` 条件影响索引使用，本质上有两层：

1. 条件形式是否能利用索引这种有序数据结构。
2. 即使能利用，优化器是否认为走索引的成本更低。

所以不能只问“这个条件能不能用索引”，还要问“这个条件会命中多少数据、会扫描多大范围、回表成本多高、优化器估算是否准确”。
