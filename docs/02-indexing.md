# 索引设计与使用

索引的核心作用是减少数据访问量。好的索引让数据库更快定位目标数据，但索引过多会增加写入成本和维护成本。

## 索引适合的场景

- 高频查询条件。
- 高选择性的过滤字段。
- 连接字段。
- 排序、分组、去重字段。
- 可以形成覆盖索引的查询。

## 联合索引原则

联合索引需要关注字段顺序。常见判断顺序：

1. 等值查询字段优先。
2. 区分度高的字段通常更靠前，但要结合查询模式。
3. 范围查询字段之后的字段，通常难以继续用于索引定位。
4. 排序字段可以放入联合索引，减少额外排序。

示例：

```sql
-- 查询模式
WHERE tenant_id = ? AND status = ? AND created_at BETWEEN ? AND ?
ORDER BY created_at DESC

-- 可能的索引
(tenant_id, status, created_at)
```

## 覆盖索引

如果查询需要的字段都能从索引中取得，就可以减少回表或额外数据页读取。

```sql
SELECT id, status, created_at
FROM orders
WHERE tenant_id = ? AND status = ?
ORDER BY created_at DESC;

-- 可考虑
(tenant_id, status, created_at, id)
```

## 索引失效常见原因

- 对索引字段使用函数或表达式。
- 隐式类型转换。
- 前导模糊匹配，例如 `LIKE '%keyword'`。
- 低选择性字段单独建索引收益有限。
- 统计信息过期导致优化器误判。

## 索引设计检查清单

- 这个索引服务哪些 SQL？
- 查询频率和性能收益是否足以抵消写入成本？
- 是否和已有索引高度重复？
- 字段顺序是否匹配过滤、排序和连接条件？
- 是否会显著增加磁盘空间和维护成本？
- 是否需要在线建索引或避开业务高峰？
