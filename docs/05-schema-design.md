# 表结构与数据模型优化

表结构决定了数据的组织方式，也会影响查询复杂度、索引大小、锁范围和存储成本。

## 字段类型选择

- 使用能表达业务含义的最小合理类型。
- 避免把数字、时间、枚举长期存成字符串。
- 金额使用定点类型，避免浮点误差。
- 时间字段统一时区和精度。
- 大字段谨慎放在高频访问表中。

## 范式与反范式

范式化可以减少冗余，保持数据一致性；反范式可以减少连接，提升读性能。

选择时需要看：

- 读写比例。
- 数据一致性要求。
- 查询复杂度。
- 数据冗余成本。
- 更新链路是否可控。

## 冷热数据

当历史数据远大于活跃数据时，可以考虑：

- 按时间分区。
- 历史表归档。
- 冷数据迁移到低成本存储。
- 查询入口区分在线查询和离线分析。

## 分区与分表

分区或分表不是万能优化，适合解决单表过大、维护成本高、冷热分离明显的问题。

需要提前评估：

- 分区键或分片键是否匹配查询条件。
- 跨分区或跨分片查询是否频繁。
- 唯一约束和事务语义是否受影响。
- 运维、备份、迁移和扩容复杂度是否可接受。

## 表结构检查清单

- 主键是否稳定且紧凑？
- 高频查询字段类型是否适合索引？
- 是否存在过宽行影响缓存命中？
- 是否有无界增长的历史数据？
- 是否有明显热点字段或热点行？
