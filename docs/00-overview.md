# 数据库优化总览

数据库优化不是单点技巧，而是一套从现象到证据、从证据到方案、从方案到验证的工程流程。

## 优化目标

常见目标包括：

- 降低查询延迟。
- 提升吞吐量。
- 降低 CPU、内存、磁盘 IO 或网络开销。
- 减少锁等待和事务冲突。
- 提升系统稳定性和可观测性。

## 基本流程

1. 明确问题：慢查询、CPU 高、IO 高、连接数耗尽、锁等待、缓存命中率低等。
2. 收集证据：慢查询日志、执行计划、等待事件、监控指标、业务流量特征。
3. 定位瓶颈：判断瓶颈在 SQL、索引、数据模型、事务、配置、硬件还是业务访问模式。
4. 制定方案：优先选择收益明确、风险较低、可回滚的方案。
5. 验证效果：对比优化前后的耗时、扫描行数、资源消耗和业务指标。
6. 固化经验：沉淀成案例、规范或自动化检查。

## 常见优化层次

| 层次 | 关注点 | 示例 |
| --- | --- | --- |
| SQL | 查询写法和执行计划 | 避免无效排序、减少回表、拆分复杂查询 |
| 索引 | 访问路径 | 联合索引、覆盖索引、选择性分析 |
| 表结构 | 数据组织方式 | 字段类型、范式与反范式、冷热数据拆分 |
| 事务 | 并发控制 | 缩短事务、降低锁范围、避免长事务 |
| 配置 | 数据库参数 | 缓冲池、连接数、工作内存、日志刷盘策略 |
| 架构 | 系统形态 | 读写分离、分库分表、缓存、异步化 |

## 优化判断原则

- 不凭感觉优化，先看指标和执行计划。
- 不把“加索引”当成唯一手段，索引也有写入和存储成本。
- 不在没有基线的情况下判断优化有效。
- 不忽略数据分布，数据量、选择性和倾斜会改变执行计划。
- 不忽略业务语义，很多性能问题源自访问模式和数据模型不匹配。
