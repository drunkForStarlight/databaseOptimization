# SQL 改写与查询调优

SQL 调优的目标是减少无效数据处理，让数据库更早、更少、更稳定地访问数据。

## 常见改写方向

- 只查询需要的列，避免无意义的 `SELECT *`。
- 把过滤条件尽量前置，让数据库更早减少数据量。
- 避免在大结果集上排序、分组和去重。
- 拆分过于复杂且优化器难以处理的 SQL。
- 用合适的连接方式替代低效子查询，或反过来根据执行计划选择子查询。
- 避免分页深翻带来的大范围扫描。

## 深分页优化

低效写法：

```sql
SELECT *
FROM orders
ORDER BY id
LIMIT 20 OFFSET 100000;
```

更适合连续翻页的写法：

```sql
SELECT *
FROM orders
WHERE id > ?
ORDER BY id
LIMIT 20;
```

## 避免无效函数过滤

低效写法：

```sql
WHERE DATE(created_at) = '2026-06-02'
```

更容易使用索引的写法：

```sql
WHERE created_at >= '2026-06-02 00:00:00'
  AND created_at <  '2026-06-03 00:00:00'
```

## 聚合优化

聚合慢通常来自扫描数据过多。可以考虑：

- 建立匹配过滤条件和分组字段的索引。
- 使用汇总表或物化视图。
- 按时间或租户拆分统计任务。
- 对实时性要求不高的指标异步计算。

## 调优验证

不要只看单次耗时。建议对比：

- 平均耗时、P95、P99。
- 扫描行数和返回行数。
- CPU、IO、内存变化。
- 执行计划是否稳定。
- 并发场景下是否引入锁等待或资源争抢。
