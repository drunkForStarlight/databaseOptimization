# SQL 执行与执行计划

执行计划用于观察数据库如何访问数据。理解执行计划是 SQL 优化的入口。

## SQL 执行的大致阶段

1. 解析 SQL：检查语法和对象权限。
2. 重写查询：做谓词下推、子查询改写、常量折叠等优化。
3. 生成执行计划：评估不同访问路径和连接顺序的成本。
4. 执行计划：读取数据、过滤、连接、排序、聚合并返回结果。

## 执行计划重点字段

不同数据库字段名称不同，但关注点相似：

- 访问类型：全表扫描、索引扫描、索引查找、范围扫描。
- 估算行数：优化器预计需要处理多少行。
- 实际行数：真实执行时处理了多少行。
- 过滤条件：哪些条件用于索引，哪些条件执行后再过滤。
- 连接方式：嵌套循环、哈希连接、合并连接。
- 排序和临时表：是否出现额外排序、临时空间或磁盘溢出。
- 成本和耗时：定位最重的执行节点。

## 阅读执行计划的顺序

1. 找最耗时或扫描行数最多的节点。
2. 看过滤条件是否有效利用索引。
3. 比较估算行数和实际行数是否偏差很大。
4. 检查连接顺序是否合理。
5. 检查排序、分组、去重是否造成额外开销。

## 常见异常信号

- 预期走索引，但实际全表扫描。
- 扫描行数远大于返回行数。
- `ORDER BY` 或 `GROUP BY` 触发大规模排序。
- 连接条件没有索引，导致大量嵌套循环。
- 估算行数严重失真，可能需要更新统计信息。

## 记录建议

每次分析慢 SQL 时，至少保留：

- 原始 SQL。
- 表结构和相关索引。
- 优化前执行计划。
- 优化后执行计划。
- 数据量、筛选条件和返回行数。
- 优化前后耗时对比。
